环境 = 场景 + MDP 组件
一个任务环境由以下组件拼接而成：

4.1 场景 (SceneCfg)
地形：平面 / 程序化生成（粗糙地形、楼梯等）
实体：机器人（robot）的 MJCF 模型 + 初始状态 + 碰撞配置
传感器：IMU、接触传感器、射线扫描、高度传感器
4.2 MDP 组件
组件	配置字段	说明
观测	observations	actor/critic 两组，每组件可配置噪声
动作	actions	关节位置 / 微分 IK 等
奖励	rewards	多个加权奖励项
终止	terminations	超时、跌倒、出界等
指令	commands	速度指令、运动指令
课程	curriculum	自适应难度调整
事件	events	域随机化（摩擦、质心、推力等）
4.3 仿真配置
物理时间步、解算器迭代次数、接触参数
GPU 并行环境数 num_envs
五、构建自定义任务的步骤
以现有的 Velocity/G1 任务为例，一个完整的新任务需要以下步骤：

第 1 步：定义 MDP 逻辑（4 个文件）
在 src/mjlab/tasks/<your_task>/mdp/ 下创建：

mdp/
├── __init__.py       # 导出所有函数
├── rewards.py        # 奖励函数
├── observations.py   # 观测函数
├── terminations.py   # 终止条件函数
└── commands.py       # (可选) 自定义指令生成器
每个函数签名遵循固定规范，例如奖励函数：
def my_reward(env: ManagerBasedRlEnv, std: float, ...) -> torch.Tensor:
    ...

第 2 步：定义环境配置工厂函数
创建 src/mjlab/tasks/<your_task>/<task>_env_cfg.py，模仿 velocity_env_cfg.py：
def make_my_task_env_cfg() -> ManagerBasedRlEnvCfg:
    return ManagerBasedRlEnvCfg(
        scene=SceneCfg(...),          # 地形 + 机器人 + 传感器
        observations={...},           # 观测组配置
        actions={...},                # 动作配置
        commands={...},               # 指令配置
        events={...},                 # 域随机化
        rewards={...},                # 奖励项
        terminations={...},           # 终止条件
        curriculum={...},             # 课程学习
        decimation=4,                 # 控制频率 = 物理频率 / decimation
        episode_length_s=20.0,        # 回合时长
    )


第 3 步：创建机器人相关的子配置
tasks/<your_task>/config/<robot_name>/
├── __init__.py       # 注册任务到 registry
├── env_cfgs.py       # 机器人特化配置
└── rl_cfg.py         # RL 训练参数
在 env_cfgs.py 中调用工厂函数并定制：
def my_robot_env_cfg(play=False) -> ManagerBasedRlEnvCfg:
    cfg = make_my_task_env_cfg()
    cfg.scene.entities = {"robot": get_my_robot_cfg()}
    # ... 覆盖传感器、奖励权重等
    return cfg


第 4 步：注册任务
在 __init__.py 中调用注册函数：
from mjlab.tasks.registry import register_mjlab_task

register_mjlab_task(
    task_id="Mjlab-MyTask-MyRobot",
    env_cfg=my_robot_env_cfg(play=False),
    play_env_cfg=my_robot_env_cfg(play=True),
    rl_cfg=my_ppo_runner_cfg(),
)
第 5 步：训练 & 评估

uv run train Mjlab-MyTask-MyRobot --env.scene.num-envs 4096
uv run play Mjlab-MyTask-MyRobot --wandb-run-path your-org/mjlab/run-id

二、dribblebot 任务本质 = 腿式机器人速度跟踪 + 足球交互

组件	dribblebot 做法	mjlab 对应
机器人	IsaacGym 加载 URDF	EntityCfg + MJCF 加载
球体	IsaacGym create_sphere API	同样用 EntityCfg + MuJoCo 几何体
传感器	14 个独立 Sensor 类	sensor/ 模块内建（ContactSensor, RayCast...）
奖励	SoccerRewards 继承 Rewards	RewardTermCfg 函数注册
观测	传感器拼接成 obs_buf	ObservationGroupCfg 管理 actor/critic 分组
域随机化	randomize_friction/restitution/... 在 step 中直接修改	EventManager 统一管理（startup/reset/interval 模式）
指令	固定速度范围，命令行反复采样	CommandManager + UniformVelocityCommandCfg
训练	内置 dribblebot_learn PPO	rsl_rl + MjlabOnPolicyRunner

2.2 核心奖励项（Dribblebot 特有）：
dribbling_robot_ball_vel — 机器人速度朝向球对齐
dribbling_robot_ball_pos — 机器人离球够近
dribbling_ball_vel — 球速度匹配指令
dribbling_robot_ball_yaw — 机器人身体朝向球

[训练阶段 - 仿真]                     [部署阶段 - 真实]
  球真实位置 (sim)                    RGB 鱼眼图像
       │                                  │
       │ + 噪声 (模拟检测误差)              │ YOLOv7 (单独训练的检测模型)
       │                                  │
       ▼                                  ▼
  球估计位置 ←──── policy πd ────→ 球估计位置
我们做的就是：给策略 ground-truth 球位置 + 噪声，模拟真实部署时 YOLO 检测的不准确性。不需要在仿真里渲染相机图像、跑物体检测——那会拖慢训练 100 倍
